# Apptopia：用 Issue 許願你想做的應用程式

這是一個以「需求表達」為核心的 GitHub 專案。任何人都可以透過建立 GitHub Issue 提出想做的工具或遊戲點子；只要需求描述足夠清楚、可驗證、可落地，我會使用我當期尚未用完的 AI 訂閱額度，在能力與資源允許的範圍內協助把需求實作成可交付的成果。

## 目標與範圍

### 這個專案想解決什麼
- 讓點子快速落地成可用的工具或可玩的遊戲原型。
- 透過 Issue 驗證「需求描述」是否足夠清楚：能否被實作、測試、交付。
- 用公開、透明的方式累積小而可用的作品。

### 範圍定義
- 作品類型：CLI 工具、Web 小工具、API 服務、桌面小程式、簡單遊戲原型、資料處理腳本等。
- 交付層級：以可運作的 MVP (最小可行產品) 為主；若需求過大，會先拆成可交付的里程碑。

## 如何參與

### 1) 提需求：建立 Issue
- 一個 Issue 只放一個需求（工具或遊戲）。
- 先搜尋是否已有相同或高度相似的 Issue；若有，請以留言補充而非重複開單。
- 需求寫得越清楚，被實作的機率越高。

### 2) 討論與澄清
- 我可能會在 Issue 留言指出缺漏、矛盾、或不可驗證的描述，並列出需要補足的資訊。
- 需求若在合理時間內仍無法補足到可實作狀態，可能會被標記為 `needs-info` 並暫停。

### 3) 實作與交付
- 若需求可落地，我會開 Pull Request (PR) 實作。
- PR 會附上：功能描述、使用方式、測試方式 (或驗收步驟)、已知限制。
- 合併後會在 Issue 回覆交付版本與使用說明，並關閉 Issue。

## 需求撰寫規格（Issue 必填）

請在 Issue 依下列格式填寫。資訊越完整，越容易被做出來。

### A. 基本資訊
- **類型**：工具 / 遊戲
- **目標使用者**：誰會用？在什麼情境用？
- **你想解決的問題**：目前痛點是什麼？

### B. 需求定義（可測試、可驗收）
- **使用流程（User Flow）**：至少 1–3 條主要流程
- **輸入/輸出**：
  - 輸入：格式、限制、範例
  - 輸出：格式、範例、錯誤訊息期望
- **功能需求（Must / Should / Could）**：
  - Must：沒有就不算完成
  - Should：有最好，但可延後
  - Could：加分項

### C. 驗收標準（Acceptance Criteria）
- 用條列列出可驗證的驗收條件（至少 3 條）
- 每條最好能用「給定/當/則（Given/When/Then）」表達

### D. 範例與測試資料（強烈建議）
- 操作範例（指令、URL、畫面流程等）
- 測試資料（可匿名或合成資料）
- 預期結果（成功與失敗各至少 1 個）

### E. 限制與偏好（可選）
- 平台：Windows / macOS / Linux / Web
- 語言/框架偏好：若有請填（不保證採用）
- 版本/環境：Node/Python/.NET 等
- 是否需要離線、是否需要儲存資料、是否需要帳號登入

## 評選與優先順序規則

我會依下列因素決定是否接單與優先順序（不代表一定會做）：

1. **可實作性**：需求是否清楚、可驗證、依賴是否合理
2. **工作量**：能否在有限的時間與可用 Tokens 範圍內完成 MVP
3. **通用性**：是否對更多人有幫助、可重用程度
4. **風險**：法律/安全/隱私/濫用風險是否可控
5. **維護成本**：未來修 bug、相依套件更新成本是否過高

## 交付標準

每個被實作的需求，原則上會包含：

- **README 或文件**：安裝、使用方式、範例
- **基本測試或驗收步驟**：至少提供可重現的測試方式
- **錯誤處理**：可理解的錯誤訊息與常見失敗情境
- **版本標記**：以 Git tag 或 release 記錄可用版本（視專案狀況）

## 變更與範圍控制

- 若 Issue 需求在討論中擴大，可能會被拆成多個 Issue 或里程碑。
- 已經合併的功能若要改動，請另開 Issue，不要在舊 Issue 追加新需求。
- 若需求與既有設計衝突，會以「保持可用與可維護」為優先。

## 行為準則

- 請保持尊重、就事論事、提供可驗證資訊。
- 惡意騷擾、釣魚、引戰、洗版、或蓄意提供危害性需求，會被關閉 Issue 並視情況封鎖。

## 安全、隱私與禁止項目

本專案拒絕實作或協助下列內容（包含但不限於）：

- 任何非法用途、侵入/破解/繞過驗證、惡意程式、竊取資料、散播惡意連結
- 針對特定個人或組織的騷擾、跟蹤、釣魚、社交工程
- 明顯侵權的內容（例如：未授權重製、破解付費、盜版）
- 涉及敏感個資蒐集或處理但沒有明確合法性與告知同意機制的需求

若需求涉及第三方服務或資料來源：
- 你必須提供該服務的公開 API/文件連結與使用限制（如速率限制、授權方式）
- 不得要求我處理你的私密金鑰、帳號密碼或任何敏感憑證

## Issue 標籤（Labels）定義

- `proposal`：新提案，尚未評估
- `needs-info`：資訊不足，等待補齊
- `accepted`：需求已可實作，排程中
- `in-progress`：開發中
- `blocked`：被依賴、環境、法律或外部因素阻擋
- `done`：已交付
- `wont-do`：不做（原因會在 Issue 說明）

## 供需與資源聲明

- 我會使用我當期尚未用完的 **AI 訂閱額度** 盡可能實作，但不承諾一定完成、也不承諾完成時間。
- 作品以「可用、可驗收」為主，不保證達到商用品質或長期維護。
- 若你願意協助：補齊需求、提供測試資料、回報 bug、寫文件、加測試，都會提高整體交付速度。
